Reporte 02: Estado para soportar 200-500 clientes

Proyecto: druvle-stock (Laravel 12)

Fecha: 11 de mayo de 2026

Objetivo: evaluar capacidad operativa y definir mejoras de performance, workers, cache y operacion.

1) Diagnostico ejecutivo

Conclusión: la app puede trabajar en ese rango con carga moderada, pero para 200-500 clientes concurrentes sostenidos necesita reforzar cache, colas, workers, observabilidad y despliegue. En el estado actual hay riesgos en picos.

Area Estado actual observado Impacto
Cache Driver por defecto en file (config/cache.php). Medio
Colas Default en database (config/queue.php) con un solo servicio de worker en docker-compose. Medio/Alto
Sesiones Driver en database (config/session.php), correcto para multi-instancia inicial. Bueno
Base de datos MySQL de un solo nodo; sin replica de lectura ni pool explicito. Medio
Infraestructura Stack docker funcional pero orientado a desarrollo (incluye vite dev). Sin Redis declarado en compose actual. Alto
Observabilidad No se evidencia stack formal de metricas/APM/trazas con alertas. Alto

2) Riesgos para 200-500 clientes

  • Cuellos en I/O por cache file: lock de disco y mayor latencia bajo concurrencia.
  • Queue database con un worker: backlog en picos de eventos/jobs.
  • Procesos pesados en request web: reportes PDF/Excel pueden competir con trafico interactivo.
  • N+1 y queries costosas: riesgo en listados con filtros/relaciones si no se perfilan.
  • DB unico nodo: sin elasticidad horizontal para lectura.

3) Plan de mejora recomendado

Fase 1 (0-3 semanas): base minima para escalar

Accion Resultado esperado Prioridad
Migrar cache a Redis (CACHE_DRIVER=redis) y definir TTL por modulo. Menor latencia en lecturas repetidas y menos carga sobre MySQL. Alta
Migrar queue a Redis (QUEUE_CONNECTION=redis) o tunear database queue + indices si no se migra aun. Procesamiento asincrono mas estable para notificaciones y tareas pesadas. Alta
Separar colas por prioridad (high, default, low) y asignar workers dedicados. Menos bloqueo entre trabajos urgentes y no urgentes. Alta
Escalar workers de 1 a 3-5 procesos segun carga y monitorear backlog. Reduccion del tiempo de espera de jobs bajo picos. Alta
Activar optimizaciones de despliegue: config/route/view cache + OPcache. Mejor tiempo de respuesta promedio. Alta

Fase 2 (1-2 meses): robustez de datos y rendimiento

  • Auditoria de indices SQL para filtros mas usados (tenant_id, fechas, estado y foraneas).
  • Instrumentar slow query log y APM para endpoints calientes.
  • Paginar y limitar payload en listados con relaciones.
  • Cachear catalogos/configuracion de baja volatilidad.
  • Mover exportaciones pesadas a jobs asincronos con notificacion al finalizar.

Fase 3 (2-4 meses): preparacion enterprise

  • Escalado horizontal de app (multiples replicas PHP-FPM detras de balanceador).
  • Replica de lectura en MySQL para consultas no transaccionales.
  • Stack formal de observabilidad (metricas, logs centralizados, trazas y alertas).
  • Definir SLOs por endpoint (latencia p95, error rate, tiempo de cola).

4) Estimacion de capacidad (orientativa)

Sin prueba de carga formal, la siguiente estimacion es referencial y depende de hardware, patron de uso y mezcla de endpoints.

Escenario Estado probable
Arquitectura actual, sin tuning operativo Apta para cargas bajas-medias, con riesgo en picos y trabajos pesados.
Con Fase 1 completa Soporte mucho mas estable para 200-500 usuarios activos con concurrencia moderada.
Con Fase 1 + 2 + pruebas de carga recurrentes Base adecuada para crecimiento sostenido y mayor predictibilidad operativa.

5) KPI de seguimiento sugeridos

  • Latencia p95/p99 por endpoint critico.
  • Tiempo en cola y backlog por tipo de job.
  • Uso de CPU/RAM por servicio (php, db, queue).
  • Tasa de errores 5xx y timeouts por minuto.
  • Top consultas SQL por tiempo total.
